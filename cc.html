Q 1(a) · CO1

What is the difference between Blockchain and Distributed Ledger Technology (DLT)?

5ML1▼
All Blockchains are DLTs, but not all DLTs are Blockchains. Blockchain is a specific subset of DLT that organizes data into cryptographically linked blocks, while DLT is the broader umbrella technology for any distributed, shared record system.
Aspect Blockchain Distributed Ledger Technology (DLT)
Definition A type of DLT that stores data in sequential, cryptographically linked blocks Any system where data is recorded, shared, and synchronized across multiple locations without a central authority
Data Structure Chain of blocks; each block contains hash of the previous block Can use any structure — DAGs (Directed Acyclic Graphs), Hashgraphs, Tangle, etc.
Relationship Is a subset/specific form of DLT Is the broader umbrella technology; Blockchain is one type
Consensus Typically uses PoW, PoS, or similar mechanisms tied to blocks Can use a wider variety of consensus models (voting, DAG-based, etc.)
Immutability Very high — altering one block requires altering all subsequent blocks Varies depending on the DLT type; not always as immutable as blockchain
Examples Bitcoin, Ethereum, Hyperledger Fabric Hedera Hashgraph, IOTA Tangle, R3 Corda, Ripple
Transparency Fully transparent on public blockchains Transparency depends on permission settings
  • DLT is the parent technology — it refers to any distributed database shared across multiple nodes without central control.
  • Blockchain adds structure — it specifically chains data into "blocks" using SHA-256 or similar hash functions, creating a verifiable chronological chain.
  • Non-blockchain DLTs like IOTA use a DAG structure where transactions validate each other directly, without grouping into blocks.
  • Blockchain is ideal for cryptocurrency because the chain structure ensures ordering and prevents double-spending without a central bank.
  • DLTs can be faster — without the block-packaging overhead, some DLT systems (e.g., Hashgraph) process transactions faster than blockchain.
  • Both eliminate intermediaries — the core shared feature is that all participants maintain a copy and changes require network consensus.
Q 1(b) · CO1

Define Blockchain and explain its types in brief.

5ML1▼
Definition: Blockchain is a decentralized, distributed, and immutable digital ledger that records transactions in a secure and transparent way across a network of computers. Data is stored in blocks linked chronologically using cryptographic hashes, making it extremely resistant to tampering or modification.
4 Types of Blockchain
  • Public Blockchain: Fully open and permissionless — anyone can join, read transaction history, and participate in consensus. Fully decentralized and transparent. Examples: Bitcoin, Ethereum.
  • Private Blockchain: Controlled by a single organization that decides who can participate and what roles they hold. Centralized, high-speed, and used for internal business processes. Example: Hyperledger Fabric.
  • Consortium Blockchain: Governed by a group of organizations rather than one entity. Permissions are set by the governing group — a balance between public and private. Used in industries like banking and supply chain. Example: R3 Corda.
  • Hybrid Blockchain: Combines features of both public and private blockchains. Certain data is kept private while specific transactions can be made public. Businesses choose which information to share openly. Example: Dragonchain.
  • Key feature of all blockchains: Each block contains a cryptographic hash of the previous block, a timestamp, and transaction data — ensuring immutability through chain linkage.
  • Immutability mechanism: Changing any block would require recalculating the hash of all subsequent blocks and gaining network consensus — practically impossible in large networks.

Q 2(a) · CO2

What consensus mechanism does Bitcoin primarily use?

5ML1▼
Bitcoin primarily uses Proof-of-Work (PoW) as its consensus mechanism. It was first described in Satoshi Nakamoto's 2008 whitepaper and remains the foundational method by which the Bitcoin network validates transactions and adds new blocks to the blockchain without any central authority.
  • Core idea: Nodes called "miners" compete to solve a complex cryptographic puzzle — specifically, finding a hash value for a new block's data that is less than a network-defined target hash (SHA-256 algorithm).
  • The "Work" element: Solving the puzzle requires massive computational power and energy. This high cost deters attackers — controlling the network would require owning 51% of total hash power (a "51% Attack"), which is prohibitively expensive.
  • Easy verification, hard to solve: Although solving the puzzle is difficult, any node on the network can instantly verify that a submitted solution is correct by simply hashing the block once.
  • Block reward incentive: The winning miner receives newly minted Bitcoin (currently 3.125 BTC after the April 2024 halving) plus all transaction fees from the block — this economic incentive ensures honest behavior.
  • Longest Chain Rule: If two miners solve the puzzle simultaneously (fork), the network adopts the chain with the most cumulative computational work — the "longest chain" — ensuring one consistent ledger.
  • Block time ~10 minutes: The network automatically adjusts the puzzle difficulty every 2,016 blocks (~2 weeks) to maintain an average block time of 10 minutes regardless of total network hash power.
  • Limitation: PoW consumes enormous amounts of electricity, which led Ethereum to switch to Proof-of-Stake (PoS) in 2022. Bitcoin continues to use PoW deliberately for its unmatched security track record.
Q 2(b) · CO2

What are "Bitcoin Scripts" and what are their applications?

5ML2▼
Bitcoin Script is a simple, stack-based, non-Turing-complete programming language embedded in every Bitcoin transaction. It defines the conditions under which funds can be spent. Every transaction contains two scripts: the scriptPubKey (locking script — sets the spending condition) and the scriptSig (unlocking script — provides the solution).
  • Stack-based execution: The Bitcoin virtual machine processes Script using a data stack. scriptSig data is pushed first, then scriptPubKey operations run. If the stack ends with a non-zero (true) value, the transaction is valid.
  • Non-Turing complete by design: Script intentionally lacks loops and complex logic. This prevents infinite execution, making it predictable and secure — critical for a financial system.
  • Application 1 — P2PKH (Standard Payments): Pay-to-Public-Key-Hash: funds sent to a Bitcoin address are unlocked only by providing the matching private key signature. This is used in ~80% of all Bitcoin transactions.
  • Application 2 — Multi-Signature (Multisig) Wallets: Requires M-of-N signatures (e.g., 2 of 3 parties must sign) before funds can be spent. Used for institutional wallets and shared accounts for enhanced security.
  • Application 3 — Time-Locked Transactions (CLTV): OP_CHECKLOCKTIMEVERIFY freezes funds until a specific block height or Unix timestamp. Foundational for the Lightning Network and inheritance arrangements.
  • Application 4 — Payment Channels (Lightning Network): Multi-sig and time-lock scripts enable off-chain instant micropayments. Only channel opening and closing are recorded on-chain — the rest is settled off-chain.
  • Application 5 — OP_RETURN (Data Embedding / Proof-of-Burn): Creates provably unspendable outputs. Used to embed small pieces of arbitrary data (e.g., document timestamps) in the blockchain or to permanently "burn" coins.
Q 3(a) · CO1

Explain the primary function of a "Wallet" in cryptocurrency terminology.

5ML1▼
A cryptocurrency wallet is a digital tool, application, or physical device used to securely store the public and private keys required to send and receive digital currencies. Importantly, it does NOT store actual cryptocurrency — the coins exist on the blockchain. The wallet simply manages the cryptographic keys that prove ownership.
  • Primary function — Key Management: The wallet's core job is storing and managing your private key (secret) and public key (shareable address). Without the private key, you cannot access or spend your cryptocurrency.
  • Public Key (Address): Like a bank account number — it can be shared with anyone who wants to send you cryptocurrency. It is derived mathematically from the private key using elliptic curve cryptography.
  • Private Key: A secret 256-bit number that authorizes transactions. Anyone with your private key controls your funds completely. "Not your keys, not your coins."
  • Seed Phrase (Recovery Phrase): A sequence of 12 or 24 English words that acts as the master backup for all private keys in a wallet. If the device is lost, the seed phrase restores the entire wallet on any device.
  • Hot Wallets (Software): Connected to the internet — convenient for frequent transactions but more vulnerable to hacking. Examples: MetaMask, Exodus, Trust Wallet (mobile apps and browser extensions).
  • Cold Wallets (Hardware): Physical devices storing private keys offline — not connected to the internet, so virtually immune to online attacks. Examples: Ledger Nano, Trezor. Best for long-term storage of large amounts.
  • Transaction signing: When sending funds, the wallet uses the private key to digitally sign the transaction — proving ownership without ever exposing the private key itself to the network.
Q 3(b) · CO1

Briefly describe what "Zero-Knowledge Systems" are used for.

5ML6▼
A Zero-Knowledge System is a cryptographic method that allows one party (the prover) to prove to another party (the verifier) that a statement is true — without revealing any information beyond the fact that the statement is true. It ensures both privacy and security simultaneously.
  • Three Core Properties: (1) Completeness — a true statement can always be proven by an honest prover; (2) Soundness — a false statement cannot be proven, except with negligible probability; (3) Zero-Knowledge — the verifier learns nothing except that the statement is true.
  • Use in Blockchain (Zcash): Zcash uses zk-SNARKs to verify that transactions are valid without revealing the sender, receiver, or amount. The network confirms math balances without seeing actual figures.
  • Use in Scaling — zk-Rollups: Ethereum uses ZK-rollups to bundle hundreds of off-chain transactions into a single proof submitted on-chain, dramatically increasing throughput while maintaining security.
  • Authentication & Identity: ZKPs enable passwordless login systems where you prove knowledge of a password without transmitting it — protecting against server-side data breaches.
  • Voting Systems: ZKPs ensure a vote was cast legitimately and counted correctly without revealing who the voter voted for — maintaining both integrity and anonymity.
  • Financial Compliance: A company can prove to a regulator that its financial position meets a threshold (e.g., "we have over $1M reserves") without disclosing exact balances — protecting competitive information.
  • Types — zk-SNARKs vs zk-STARKs: SNARKs have small proof sizes and fast verification but require a trusted setup. STARKs are larger but quantum-resistant and require no trusted setup — important for future-proofing.

List and explain the stages in a typical Bitcoin transaction lifecycle.

5ML3▼
A Bitcoin transaction lifecycle is the complete journey of a transaction from the moment a user initiates a transfer until it becomes permanently recorded on the blockchain.

6 Confirmations = Irreversible (TX permanently settled)
  • Creation: The wallet selects UTXOs (Unspent Transaction Outputs) as inputs and creates a new transaction specifying recipient address, amount, and change address.
  • Digital Signing: The sender's private key mathematically signs the transaction using ECDSA — proving ownership without revealing the private key.
  • Broadcasting & Mempool: The signed transaction is broadcast to the peer-to-peer network. Each node validates and stores it in the mempool (memory pool) of pending transactions.
  • Mining: Miners select pending transactions (prioritizing higher fees) and bundle them into a candidate block. They then compete using PoW to find a valid hash below the target.
  • Confirmation: Once a block is mined and accepted, the transaction has 1 confirmation. Each subsequent block adds another confirmation. 6 confirmations (~60 mins) is the standard for high-value transactions to be considered irreversible.
  • Irreversibility: Once deeply embedded in the chain, reversing a transaction would require re-mining all subsequent blocks with more than 50% of the network's hash power — economically infeasible.
Q 4(b) · CO2

Name two techniques used for Bitcoin scalability and performance improvement.

5ML6▼
Bitcoin's base layer (Layer 1) can only process ~3–7 transactions per second (TPS) — far below Visa's ~24,000 TPS. Two major solutions address this scalability problem:
Technique 1: Segregated Witness (SegWit) — Layer 1 Solution
  • Activated in August 2017 as a soft fork to the Bitcoin protocol.
  • Separates signature data (the "witness" data) from the main transaction data in each block, freeing up block space.
  • Effectively increases block capacity from ~1MB to ~4MB equivalent (using "block weight" units), allowing more transactions per block without changing the 1MB size limit.
  • Also fixed transaction malleability — a bug where txid could be changed before confirmation, which was required before Lightning Network could work.
Technique 2: Lightning Network — Layer 2 Solution
  • Payment channel network built on top of Bitcoin that enables instant, near-zero-fee transactions between parties.
  • Opening a channel: Two parties lock funds in a multisig address on-chain. They can then transact unlimited times off-chain by exchanging signed states.
  • Only two on-chain transactions: Opening the channel and closing/settling the final balance. All intermediate transactions stay off-chain, massively reducing blockchain load.
  • Network routing: Payments can route through multiple intermediate nodes, so you don't need a direct channel with everyone — just a path through the network.
  • Impact: Lightning Network theoretically enables millions of TPS and is used for Bitcoin micropayments (e.g., El Salvador's Bitcoin payments ecosystem).

Q 1(a) · CO3

Define Ethereum and state three differences between Ethereum and Bitcoin.

5ML1▼
Definition: Ethereum is a decentralized, open-source blockchain platform that enables the creation and execution of Smart Contracts and Decentralized Applications (dApps), with its native cryptocurrency Ether (ETH) used to pay for computation and transactions. While Bitcoin is "digital gold," Ethereum is often called "digital oil" — the fuel powering a programmable blockchain economy.
Aspect Bitcoin (BTC) Ethereum (ETH)
Primary Purpose Digital currency and store of value ("digital gold") Programmable platform for smart contracts and dApps ("digital oil")
Consensus Mechanism Proof-of-Work (PoW) — uses SHA-256 mining since 2009 Proof-of-Stake (PoS) — switched from PoW in "The Merge" (September 2022)
Supply Cap Fixed maximum of 21 million BTC — deflationary by design No fixed cap; dynamic issuance with fee burning (EIP-1559) — can be deflationary
Block Time ~10 minutes per block ~12 seconds per block — much faster
Programmability Very limited — Bitcoin Script is simple and non-Turing-complete Fully Turing-complete — Solidity language enables complex smart contracts
Transaction Speed ~7 transactions per second (TPS) ~15–30 TPS (higher with Layer 2)
Energy Consumption Very high — PoW requires massive electricity Very low — PoS reduced energy use by ~99.95%
  • Difference 1 — Purpose: Bitcoin was designed as a peer-to-peer electronic cash system to replace traditional money. Ethereum was designed as a world computer — a platform for building decentralized applications, not just transferring value.
  • Difference 2 — Smart Contracts: Ethereum introduced smart contracts — self-executing code stored on the blockchain. Bitcoin has very limited scripting capability and cannot run complex programs.
  • Difference 3 — Consensus: Bitcoin uses energy-intensive Proof-of-Work with SHA-256 hardware miners competing globally. Ethereum uses Proof-of-Stake where validators lock up 32 ETH as collateral to propose and attest to blocks — 99.95% more energy efficient.
Q 1(b) · CO3

What is MetaMask's Roles in Smart Contracts?

5ML1▼
MetaMask is a browser extension and mobile wallet that acts as the critical bridge between a user's web browser and the Ethereum blockchain. In the context of smart contracts, it simultaneously serves as a secure key vault, transaction signer, and network provider.
  • Role 1 — Wallet & Key Management: MetaMask securely stores the user's private keys in an encrypted vault within the browser. It generates and manages Ethereum accounts (EOAs) without ever exposing private keys to websites.
  • Role 2 — Transaction Signing for Deployment: When a developer deploys a smart contract, MetaMask intercepts the transaction, shows the user the gas cost, and prompts for approval. The contract's compiled bytecode is then sent to the network only after the user signs.
  • Role 3 — Write Transactions (State-Changing Calls): Functions that modify blockchain state (e.g., transfer tokens, update a variable) require a MetaMask signature and gas fee payment. MetaMask presents a clear popup showing what the transaction will do.
  • Role 4 — Read Calls (Free, No Signature): Functions that only read data from a contract (marked view or pure in Solidity) can be called for free without any MetaMask interaction — no gas, no popup.
  • Role 5 — Provider Service (window.ethereum): MetaMask injects a global JavaScript object called window.ethereum into every webpage. Developer tools (Remix IDE, Hardhat, ethers.js) use this to communicate with the blockchain — sending transactions, reading state, and detecting network/account changes.
  • Role 6 — Network Switching: Developers can switch MetaMask between mainnet, testnets (Sepolia, Goerli), or custom networks. This allows testing smart contracts on free test Ether before deploying on the expensive main network.
  • Role 7 — dApp Authentication: MetaMask enables "Sign in with Ethereum" — users prove ownership of their wallet address as an identity, replacing username/password systems with cryptographic proof.

Q 2(b) · CO3

Write a short note on ERC-20

5ML2▼
ERC-20 (Ethereum Request for Comments 20) is a technical standard that defines a set of rules for creating fungible tokens on the Ethereum blockchain. Proposed by Fabian Vogelsteller in November 2015, it is the foundation of the vast majority of tokens in the crypto ecosystem — from stablecoins like USDT to governance tokens like UNI.
6 Mandatory Functions of ERC-20
Function Description
totalSupply() Returns the total number of tokens in existence
balanceOf(address) Returns the token balance of a specific wallet address
transfer(to, amount) Transfers tokens from the caller's account to another address
transferFrom(from, to, amount) Transfers tokens on behalf of another user (requires prior approval)
approve(spender, amount) Grants permission to another address to spend a specified amount
allowance(owner, spender) Returns the remaining amount a spender is allowed to withdraw
  • Standardization: ERC-20 ensures all tokens behave identically — any wallet, exchange, or application that supports ERC-20 can instantly work with any compliant token without custom integration.
  • High Liquidity: Because all ERC-20 tokens follow the same interface, they can be listed on decentralized exchanges (like Uniswap) immediately after deployment, providing instant market access.
  • ICO Enablement: The ERC-20 standard powered the 2017 ICO boom — startups could create and distribute tokens on Ethereum in hours, raising millions in funding.
  • Events — Transfer & Approval: ERC-20 also defines two mandatory events (Transfer, Approval) that allow frontends and indexers to efficiently track token movements without querying the full blockchain.
  • Limitations: ERC-20 has known issues — tokens accidentally sent directly to a contract address are permanently lost (addressed by ERC-777 and ERC-223 improvements). It also doesn't handle non-fungible assets (that's ERC-721/NFTs).
  • Examples: USDT (Tether), USDC, LINK (Chainlink), UNI (Uniswap), SHIB (Shiba Inu) are all ERC-20 tokens running on Ethereum.

Q1

Define cryptocurrency and explain types of blockchain with use cases.

10M▼
Cryptocurrency: A digital currency in which transactions are verified and records maintained by a decentralized system using cryptography, rather than by a centralized authority. Examples: Bitcoin (BTC), Ethereum (ETH), Tether (USDT), XRP.
4 Types of Blockchain
  • Public Blockchain: Open to anyone — fully decentralized, transparent, permissionless. Anyone can read, write, and participate in consensus. Examples: Bitcoin, Ethereum. Use case: Digital currency, public finance, open NFT markets.
  • Private Blockchain: One organization controls all access and roles. High speed, low latency, controlled privacy. Example: Hyperledger Fabric. Use case: Enterprise internal processes — supply chain tracking, employee records, internal audits.
  • Consortium Blockchain: Governed by a group of organizations — balanced between public and private. Semi-decentralized — members share governance. Example: R3 Corda, Quorum. Use case: Banking consortiums, cross-industry logistics, inter-bank settlements.
  • Hybrid Blockchain: Combines private and public features — some data is private, specific transactions can be made public. Flexible governance. Example: Dragonchain, IBM Food Trust. Use case: Healthcare (private patient records + public drug traceability), real estate.
  • Key properties of all blockchains: Immutability (data cannot be altered once written), Transparency (participants can audit), Decentralization (no single point of control), Consensus-driven (changes require network agreement).
  • Coins vs Tokens: Coins (BTC, ETH) operate on their own blockchain and function as digital money. Tokens are built on existing blockchains (ERC-20 on Ethereum) and represent assets, rights, or utility within a specific ecosystem.
Q2

What is Blockchain? Explain DLT and the relationship between them.

10MCommon: CIE-1 Q1(a) + Q1(b)▼
Blockchain: A decentralized, distributed, and immutable digital ledger recording transactions across a network of computers in blocks cryptographically linked in sequence. DLT: Distributed Ledger Technology is any shared, synchronized digital record system maintained across multiple nodes without a central administrator.
  • DLT fundamentals: Each node in the peer-to-peer network maintains an identical, synchronized copy of the ledger. Updates are processed independently by each node and verified collectively through a consensus mechanism before being applied network-wide.
  • Key DLT characteristics: Decentralized (no single entity controls data), Transparent & Auditable (clear audit trail), Secure (advanced cryptographic techniques ensure integrity), Efficient (reduces intermediaries, lowers costs).
  • Relationship — Blockchain is a subset of DLT: All blockchains are DLTs, but not all DLTs are blockchains. Blockchain is the most well-known DLT implementation, first popularized by Bitcoin.
  • What makes Blockchain unique among DLTs: Data is grouped into discrete "blocks." Each block contains a cryptographic hash (SHA-256) of the previous block, creating a chronological chain. Altering one block requires altering every subsequent block — making tampering practically impossible.
  • Non-blockchain DLTs: IOTA uses DAG (Directed Acyclic Graph) — transactions validate each other directly without blocks. Hedera Hashgraph uses "gossip about gossip" consensus with virtual voting — much faster than blockchain.
  • Consensus Mechanism in DLT: Without a central authority, all nodes must agree on the valid state of the ledger. This is achieved through mechanisms like PoW (Bitcoin), PoS (Ethereum), or PBFT (Hyperledger) — each with different security/speed trade-offs.
Q3

Explain the elements of Distributed Computing in detail.

10M▼
Distributed Computing is a system where components on networked computers communicate and coordinate their actions to achieve a common goal. The system appears as a single unified computer to the end user, even though it runs on multiple independent machines.
  • Nodes (Independent Computers): Each node is an autonomous computer with its own processor and memory. Nodes can function as clients (requesting services), servers (providing services), or both (peer-to-peer systems like Bitcoin). Each node can fail independently without bringing down the entire system.
  • Communication Infrastructure (Message Passing): The network connecting nodes allows them to communicate by sending and receiving messages. In blockchain, transactions and blocks are propagated through the network using a gossip protocol — each node shares new information with its neighbors, spreading it exponentially.
  • Distributed Data Storage: Data is stored and managed across multiple machines rather than in one central location. This creates redundancy — if one node fails, data isn't lost. In blockchain, every full node stores a complete copy of the entire blockchain (Bitcoin blockchain is ~600GB as of 2024).
  • Coordination and Control: Software and consensus algorithms manage the distributed components, synchronize actions, and ensure coherent operation without a shared clock or memory. In Bitcoin, PoW serves as the coordination mechanism — the longest chain rule resolves conflicts.
  • Fault Tolerance: Distributed systems are designed to continue operating even when some nodes fail. Byzantine Fault Tolerance (BFT) handles even malicious nodes — a system is BFT if it functions correctly as long as fewer than 1/3 of nodes are faulty or malicious.
  • Transparency vs Privacy: In public distributed systems (Bitcoin), all data is visible to all nodes. In permissioned systems (Hyperledger), privacy is maintained by only sharing relevant data with authorized nodes.
  • No global clock: One key challenge in distributed computing is that nodes cannot perfectly synchronize their clocks. Blockchain solves this using block timestamps + longest chain rule — providing an agreed-upon ordering of events without requiring synchronized clocks.
Q4

Describe Cryptographic Hashing and its role in Blockchain security.

10M▼
A hash function is a mathematical algorithm that transforms any input data (of any size) into a fixed-size output called a hash value, hash digest, or simply a "hash." In blockchain, SHA-256 (producing a 256-bit/64-character hex hash) is the primary hash function used by Bitcoin.
  • Deterministic: The same input always produces the exact same hash output. This allows anyone to verify data integrity — if the hash of received data matches the expected hash, the data is unchanged.
  • One-Way (Pre-image Resistance): Given a hash output, it is computationally infeasible to reverse-engineer the original input. This is why passwords can be stored as hashes — even if the database is stolen, passwords cannot be recovered.
  • Collision Resistance: It is practically impossible to find two different inputs that produce the same hash output. This ensures each block of data has a unique fingerprint.
  • Avalanche Effect: Changing even a single character in the input produces a completely different, unpredictable hash. For example, "Hello" and "hello" produce entirely different hashes. This prevents any gradual manipulation of data.
  • Fixed-Length Output: Regardless of input size (1 byte or 1TB), the hash is always 256 bits (for SHA-256). This allows efficient comparison and storage.
  • Role in Blockchain — Block Linking: Each block header contains the hash of the previous block. If any past block's data is changed, its hash changes, breaking the link to the next block, which then has an invalid previous hash — invalidating all subsequent blocks.
  • Role in Mining (PoW): Miners repeatedly hash block header data with different nonce values, trying to find a hash below the network's target difficulty. This process (searching ~10^22 hashes) is the "work" in Proof-of-Work.
  • Merkle Trees: Transaction data within a block is organized into a Merkle Tree — a binary tree of hashes. The Merkle Root (hash of all transaction hashes) is stored in the block header, allowing efficient verification of individual transactions without downloading the entire block.
Q5

Explain Cryptographic Signature Schemes — Key Generation, Signing, and Verification.

10M▼
A cryptographic signature scheme allows a sender to prove the authenticity and integrity of a message using their private key. The signature can be verified by anyone using the corresponding public key, without needing to know the private key. Bitcoin and Ethereum both use ECDSA (Elliptic Curve Digital Signature Algorithm).
3 Core Algorithms
  • Algorithm 1 — Key Generation: Generates a key pair: a private key (a random 256-bit number — kept strictly secret, used only for signing) and a corresponding public key (derived from the private key using elliptic curve multiplication — shared openly for verification). In Bitcoin, the public key is further hashed to produce the wallet address.
  • Algorithm 2 — Signing (Sign): The signer uses their private key and the message (or its SHA-256 hash) to compute a unique digital signature. The signature is a mathematical pair (r, s) in ECDSA. Even a tiny change to the message produces a completely different signature — ensuring integrity.
  • Algorithm 3 — Verification (Verify): Anyone can use the signer's public key, the original message, and the signature (r, s) to verify that the signature is genuine — without knowing the private key. Returns TRUE if valid, FALSE if tampered.
  • Non-repudiation: Since only the holder of the private key can produce a valid signature, the signer cannot deny having signed the message — providing legal-grade proof of authorization.
  • Application in Bitcoin: Every Bitcoin transaction must be signed with the sender's private key using ECDSA. Miners verify the signature before including the transaction in a block — ensuring only the rightful owner can spend the funds.
  • Elliptic Curve advantage: ECDSA provides the same security level as RSA with much shorter key lengths — 256-bit ECC ≈ 3072-bit RSA security. This means smaller transaction sizes and faster verification on the blockchain.
  • Schnorr Signatures (Taproot, 2021): Bitcoin's Taproot upgrade introduced Schnorr signatures as an alternative to ECDSA — they are more efficient, support native multi-signature aggregation (multiple signatures become one), and reduce transaction size.

Q7

What is Elliptic Curve Cryptography (ECC)? Explain its applications in blockchain.

10M▼
Elliptic Curve Cryptography (ECC) is a form of asymmetric cryptography based on the algebraic structure of elliptic curves over finite fields. It provides the same level of security as RSA with significantly shorter key lengths, making it ideal for resource-constrained environments like mobile devices and blockchain networks.
  • Mathematical basis: An elliptic curve is defined by the equation y² = x³ + ax + b over a finite field. The security relies on the Elliptic Curve Discrete Logarithm Problem (ECDLP) — given points P and Q = kP on the curve, finding k (the private key) is computationally infeasible.
  • Key size advantage: 256-bit ECC key provides security equivalent to a 3,072-bit RSA key. Shorter keys mean: smaller transaction sizes on blockchain, faster verification, lower computational overhead, and less storage per transaction.
  • Application in Bitcoin — ECDSA: Bitcoin uses ECDSA with the secp256k1 curve. Each Bitcoin wallet generates a 256-bit private key → derives a 512-bit public key via elliptic curve multiplication → hashes it (SHA-256 + RIPEMD-160) to produce the 160-bit Bitcoin address. All transactions are signed with ECDSA.
  • Application in Ethereum: Ethereum also uses ECDSA with secp256k1 for signing transactions. Ethereum addresses are derived from the last 20 bytes of the Keccak-256 hash of the public key. Every transaction requires ECDSA signature verification by the network.
  • One-way trapdoor function: Public key = private key × Generator Point G (a fixed point on the curve). This multiplication (point doubling and adding) is easy to compute forward but practically impossible to reverse — the entire security model rests on this property.
  • Schnorr Signatures (Bitcoin Taproot 2021): An alternative to ECDSA using the same secp256k1 curve but with a simpler, more efficient algorithm. Key advantage: multiple signatures can be aggregated into one (MuSig), enabling more private and efficient multi-signature transactions.
  • Why not RSA for blockchain? RSA requires ~3,072-bit keys for equivalent security — this would make Bitcoin transactions ~10× larger, requiring much more blockchain storage space, and verification would be slower, increasing computational load on every full node.
Q8

Define Verifiable Random Function (VRF) and explain its role in blockchain applications.

10M▼
A Verifiable Random Function (VRF) is a cryptographic function that takes inputs (including a private key), computes a pseudorandom output, and produces a proof of authenticity that anyone can verify. It combines randomness with verifiability — two properties that don't naturally coexist.
  • Three outputs of a VRF: (1) Random Output — a pseudorandom number derived from the inputs; (2) Proof — a cryptographic proof that the output was computed correctly using the specified private key; (3) Verification — anyone with the public key can verify the proof is valid without access to the private key.
  • Why "verifiable" randomness matters in blockchain: In decentralized systems with no trusted authority, any source of randomness could be manipulated. VRFs ensure the random output was generated honestly and deterministically from known inputs — neither predictable in advance nor manipulable after the fact.
  • Application 1 — Validator/Leader Selection (Algorand, Cardano): PoS blockchains use VRFs to pseudorandomly select which validator proposes the next block. The selected validator proves they were legitimately chosen by publishing their VRF proof — other nodes verify without any central coordinator.
  • Application 2 — Chainlink VRF (Smart Contract Randomness): Chainlink's VRF service provides provably fair on-chain randomness for smart contracts. Applications: NFT trait randomization, lottery draws, gaming (fair dice rolls), random assignment — all provably unmanipulated.
  • Application 3 — Rare NFTs: NFT projects use VRFs to assign rare traits during minting. Without VRF, the NFT creator or miners could manipulate which wallet receives rare items. VRF proof published on-chain proves the rarity assignment was fair.
  • Application 4 — Unbiased Outcomes in DeFi: Protocol parameter randomization, random order execution to prevent front-running, and random sampling for security audits all benefit from VRF-based randomness that neither the protocol team nor miners can influence.
  • VRF vs simple on-chain randomness: Using block hash as randomness source is insecure — miners can selectively withhold blocks to get favorable random outputs. VRF removes this attack vector by generating randomness off-chain (but verifiably) before it influences the chain outcome.
Q9

Explain Zero-Knowledge Systems — properties, types (zk-SNARKs vs zk-STARKs), and applications.

10MCommon: CIE-1 Q3(b)▼
A Zero-Knowledge Proof is a cryptographic method allowing a prover to convince a verifier that a statement is true without revealing any information beyond that truth. The classic example: proving you know a password without transmitting the password itself.
  • Property 1 — Completeness: If the statement is true and both parties are honest, the verifier will always be convinced. A legitimate prover with valid knowledge will always be able to produce a valid proof.
  • Property 2 — Soundness: If the statement is false, no cheating prover can convince an honest verifier (except with negligible probability). The system catches liars with overwhelming probability.
  • Property 3 — Zero-Knowledge: The verifier learns nothing from the interaction except that the statement is true. The proof reveals zero information about the underlying secret.
  • zk-SNARKs: Zero-Knowledge Succinct Non-Interactive Arguments of Knowledge. Small proof size (hundreds of bytes), millisecond verification, single published proof. Require a trusted setup (a one-time initialization ceremony). Used in Zcash and Ethereum zk-rollups.
  • zk-STARKs: Zero-Knowledge Scalable Transparent Arguments of Knowledge. No trusted setup required — transparent initialization. Quantum-resistant. Larger proof sizes than SNARKs. Used in StarkWare's scaling solutions for Ethereum.
  • Application — Blockchain/Crypto: Zcash uses zk-SNARKs for shielded transactions. Ethereum zk-rollups (Polygon zkEVM, zkSync) use ZKPs to bundle thousands of transactions and submit one proof on-chain — dramatically improving scalability.
  • Application — Authentication: ZKPs enable passwordless login — you prove knowledge of a secret without sending it over the network. Zero-knowledge password proofs prevent server-side data breaches from exposing passwords.
  • Application — Voting, Finance, Supply Chain: Prove a vote is valid without revealing who you voted for. Prove financial compliance without disclosing exact figures. Verify product provenance without revealing trade secrets.

Q12

Explain Distributed Consensus using Proof-of-Work (PoW) in Bitcoin.

10MCommon: CIE-1 Q2(a)▼
Distributed Consensus in Bitcoin solves the fundamental problem: how can thousands of untrusting nodes around the world agree on one true version of transaction history without any central authority? Proof-of-Work answers this by transforming trust into a computational and economic challenge.
  • The Problem: In a decentralized network, any node can broadcast conflicting transactions (e.g., double-spending the same coin). Without a trusted authority, the network needs a mechanism to agree on which transaction history is legitimate.
  • Mining Competition: Nodes called "miners" collect pending transactions and compete to solve a cryptographic puzzle: find a nonce value such that SHA-256(SHA-256(block_header)) produces a hash below the current network target difficulty. This requires trillions of hash computations.
  • Costly Work = Security: Solving the puzzle requires enormous computational resources. To attack the network (rewrite history), an attacker would need >50% of the total global hash power — currently equivalent to billions of dollars of hardware and electricity. This makes attacks economically irrational.
  • Easy Verification: While solving takes minutes and enormous energy, verifying a solution takes microseconds — any node simply runs the hash function once. This asymmetry is essential for network scalability.
  • Longest Chain Rule: When two miners find valid solutions simultaneously (a fork), nodes temporarily have two valid chain heads. The network converges on the chain with the most cumulative PoW — the "heaviest" chain. The other chain is orphaned, and its transactions return to the mempool.
  • Block Reward Incentive: The winning miner receives: (a) Block subsidy — newly created BTC (currently 3.125 BTC); (b) All transaction fees from included transactions. This economic incentive ensures rational actors mine honestly rather than attack the network.
  • Difficulty Adjustment: Every 2,016 blocks (~2 weeks), the network recalculates the target difficulty based on how fast the last 2,016 blocks were found. If blocks came faster than 10 minutes, difficulty increases; slower, it decreases. This maintains the ~10-minute average block time regardless of total network hash power.
  • Finality: A transaction is not truly "final" in PoW — technically it can be reversed if someone mines a longer chain. However, after 6 confirmations (~1 hour), reversing would require re-mining 6 blocks while the rest of the network continues mining — statistically impossible with honest majority.

Restoring validity would require re-mining all subsequent blocks faster than the entire honest network — practically impossible.

Explain Bitcoin Script Execution in detail — with P2PKH step-by-step example.

10MCommon: CIE-1 Q2(b)▼
Bitcoin Script is a simple, stack-based, non-Turing-complete language used to verify that the conditions for spending a transaction's funds have been met. Every transaction contains two scripts that must be combined and executed successfully for the transaction to be valid.
Two Scripts in Every Transaction
  • scriptPubKey (Locking Script): Attached to a transaction output (UTXO). Defines the conditions that must be met to spend those funds. Written by the sender when they create the transaction. Acts as a "digital lock."
  • scriptSig (Unlocking Script): Provided in the spending transaction's input. Contains the data (signature, public key) needed to satisfy the locking script. Written by the spender. Acts as the "digital key."
  • Execution order: (1) scriptSig is executed first — pushes its data onto an empty stack; (2) scriptPubKey is executed — uses data from the stack; (3) If execution completes without error and leaves TRUE on top of the stack → transaction is VALID; otherwise → REJECTED.
P2PKH (Pay-to-Public-Key-Hash) Step-by-Step
  • Scripts: scriptSig = <Signature> <PublicKey>  |  scriptPubKey = OP_DUP OP_HASH160 <PubKeyHash> OP_EQUALVERIFY OP_CHECKSIG
  • Step 1: <Signature> pushed onto stack. <PublicKey> pushed onto stack. Stack: [Sig, PubKey]
  • Step 2 — OP_DUP: Duplicates the top item (PublicKey). Stack: [Sig, PubKey, PubKey]
  • Step 3 — OP_HASH160: Hashes the top PublicKey with SHA-256 then RIPEMD-160 → produces PubKeyHash. Stack: [Sig, PubKey, PubKeyHash_derived]
  • Step 4: The expected <PubKeyHash> from the locking script is pushed onto the stack. Stack: [Sig, PubKey, PubKeyHash_derived, PubKeyHash_expected]
  • Step 5 — OP_EQUALVERIFY: Compares the two PubKeyHashes. If they don't match → script ABORTS (wrong key). If they match → both are removed from stack. Stack: [Sig, PubKey]
  • Step 6 — OP_CHECKSIG: Verifies the Signature against the PubKey and the transaction data. If valid → pushes TRUE → transaction VALID. If invalid → FALSE → transaction REJECTED.
Q17

Discuss the Applications of Bitcoin Scripts.

10MCommon: CIE-1 Q2(b)▼
Bitcoin Script, despite being intentionally simple and non-Turing-complete, enables a surprisingly diverse range of financial applications by defining flexible conditions under which funds can be spent.
  • Standard Payments (P2PKH): Pay-to-Public-Key-Hash — funds sent to a Bitcoin address are unlocked only by the matching private key signature. Used in ~80% of all Bitcoin transactions. Simple, fast, and widely supported by all wallets.
  • Multi-Signature (Multisig) Wallets: Requires M-of-N signatures before funds can be spent (e.g., 2 of 3 parties must agree). Used for institutional custody, corporate wallets, and escrow. Significantly enhances security — no single private key loss can drain the wallet.
  • Time-Locked Transactions (CLTV — OP_CHECKLOCKTIMEVERIFY): Funds are frozen until a specified future block height or Unix timestamp. Used for: inheritance arrangements (funds unlockable after a date), employee vesting schedules, and as the foundational building block for the Lightning Network.
  • Payment Channels (Lightning Network): Multi-sig + time-lock scripts together enable bidirectional payment channels — two parties can transact unlimited times off-chain, with the final balance settled on-chain. Enables instant micropayments with near-zero fees.
  • Escrow Services (2-of-3 Multisig): Three parties: buyer, seller, and arbitrator. Funds released only when 2 of 3 parties sign — either buyer+seller agree the deal is complete, or buyer/seller+arbitrator resolve a dispute. Trustless escrow without a bank.
  • Proof-of-Burn (OP_RETURN): Creates provably unspendable outputs by including OP_RETURN — coins are permanently destroyed. Used for: creating scarcity, bootstrapping alternative cryptocurrencies (altcoins verified against Bitcoin's chain), and reducing token supply.
  • Data Embedding (OP_RETURN): OP_RETURN allows embedding up to 80 bytes of arbitrary data in a transaction without affecting the UTXO set. Used for: document timestamping (proof of existence on a specific date), the Ordinals protocol (Bitcoin NFTs), and cross-chain communication protocols.
Q18

Explain Bitcoin Network Architecture — Core Components and 5 Functional Layers.

10M▼
The Bitcoin network is a decentralized, peer-to-peer (P2P) system with a flat topology — no central server or authority controls it. It operates using a combination of a public distributed ledger (blockchain), PoW consensus, and various node types to maintain security and integrity.
Core Components
  • Peer-to-Peer Network: Nodes (computers running Bitcoin software) connect directly to each other in a mesh network — acting as both clients and servers simultaneously. Transactions and blocks are relayed on a best-effort basis. No single node is critical to network operation.
  • Blockchain (Distributed Ledger): A public, immutable, chronologically ordered database recording every Bitcoin transaction since the genesis block (January 3, 2009). Each block's hash is dependent on the previous — creating tamper-evidence through cryptographic linkage.
  • Full Nodes: Store a complete copy of the entire blockchain, independently verify every transaction and block against all consensus rules. Currently ~15,000+ reachable full nodes globally. They are the guardians of Bitcoin's rules — they reject invalid transactions regardless of miner activity.
  • SPV Nodes (Lightweight): Used on devices with limited storage (smartphones). Download only block headers (~80 bytes vs ~1MB per block) and rely on full nodes for transaction verification — trading some security for convenience.
  • Miners: Specialized nodes that compete to solve PoW puzzles using ASICs (specialized chips). They bundle transactions, create new blocks, and receive block rewards. Currently ~1-2 million ASIC mining machines globally.
5 Functional Layers
  • Layer 1 — Hardware/Infrastructure: Physical computers, mining ASICs (SHA-256 chips), GPUs, and storage. The computational power and storage foundation the entire network is built upon.
  • Layer 2 — Data Layer: Manages data structure and storage. Blocks, cryptographic hashes (SHA-256), Merkle Trees (organizing transactions), digital signatures (ECDSA), UTXOs. Defines what data exists and how it's structured.
  • Layer 3 — Network Layer: Facilitates P2P communication between nodes. Handles node discovery (DNS seeding), transaction propagation (gossip protocol), block propagation (Compact Block Relay — reduces bandwidth by 80%), and peer management.
  • Layer 4 — Consensus Layer: Defines rules for network agreement. Proof-of-Work (SHA-256) algorithm, the Longest Chain Rule, difficulty adjustment, block reward rules, and all validation logic that every full node enforces independently.
  • Layer 5 — Application Layer: End-user interfaces. Wallet software (MetaMask, Ledger), exchanges, payment processors, Lightning Network clients, and any application allowing users to interact with the underlying Bitcoin network.
Q19

Discuss the Bitcoin Scalability Problem and its Layer 1 & Layer 2 solutions.

10MCommon: CIE-1 Q4(b)▼
Bitcoin's design intentionally limits throughput to ~3–7 transactions per second (TPS) on its base layer (Layer 1) — compared to Visa's ~24,000 TPS. This is an intentional trade-off: prioritizing decentralization and security over speed. Various solutions have been developed across two categories.
The Core Constraints
  • Block Size Limit: ~1–4MB per block (post-SegWit). Only so many transactions fit per block.
  • Block Time: ~10 minutes per block. Even if blocks were full, throughput is limited by time between blocks.
  • Combined result: ~7 TPS max on-chain. During 2017 and 2021 bull runs, fees spiked to $50+ per transaction due to mempool congestion.
Layer 1 Solutions (On-Chain)
  • SegWit (Segregated Witness, 2017): Separates digital signature data (witness data) from the main transaction block. This effectively increases block capacity to ~4MB equivalent (measured in "weight units"), fitting more transactions per block. Also fixed the transaction malleability bug, enabling Lightning Network.
  • Taproot (2021): Introduced Schnorr signatures — more efficient than ECDSA. Multiple-signature transactions (multisig) appear identical to single-sig transactions, saving space. MAST (Merkelized Abstract Syntax Trees) reduces size of complex scripts.
Layer 2 Solutions (Off-Chain)
  • Lightning Network: Creates bidirectional payment channels between parties. Funds are locked on-chain in a multisig address; parties transact unlimited times off-chain by exchanging signed state updates. Only two on-chain transactions ever occur (open + close). Theoretically millions of TPS. Used for Bitcoin micropayments globally.
  • Sidechains (Liquid Network, RSK): Independent blockchains running parallel to Bitcoin with a two-way peg. Assets are locked on Bitcoin mainnet and mirrored on the sidechain, which can operate faster or with different rules. Settlement returns to the mainnet.
  • Rollups: Bundle hundreds of off-chain transactions into one batch with a single validity proof submitted on-chain. Inherits Bitcoin's security for final settlement while massively increasing throughput.
  • Network-level optimizations: Compact Block Relay reduces bandwidth for propagating new blocks. Erlay reduces the network overhead of transaction announcement between nodes.

Compare Payment Channels and State Channels. Explain the Lightning Network in detail.

10M▼
Payment Channels are a specialized type of State Channel. State Channels are the broader concept — any off-chain interaction governed by smart contracts. Payment Channels are specifically designed for transferring value.
Aspect Payment Channels State Channels
Focus Transferring value (money/cryptocurrency) between parties Any off-chain interaction: payments, gaming, data updates, app state
Example Bitcoin Lightning Network, Raiden Network (ETH) Ethereum State Channels, CounterFactual, Perun
Mechanism Parties lock funds on-chain; exchange signed balance updates off-chain Lock any smart contract state on-chain; execute complex logic off-chain
Smart Contract Logic Simple value transfer conditions Any arbitrary logic — turn-based games, multi-party agreements, data feeds
On-chain activity Only channel open + close transactions Only channel open + final state submission
Closing Final balance broadcast on-chain; parties receive their funds Final state after all off-chain operations submitted to the chain
Lightning Network — Detailed Explanation
  • Opening a channel: Alice and Bob both sign a multisig transaction locking some BTC on-chain (e.g., Alice 0.1 BTC, Bob 0.1 BTC). This opening transaction is broadcast to and confirmed by the Bitcoin mainchain.
  • Off-chain transactions: Alice and Bob exchange signed balance update messages directly — no blockchain. If Alice pays Bob 0.01 BTC, they both sign a new state: "Alice=0.09 BTC, Bob=0.11 BTC." This happens instantly and for free.
  • Invalidation mechanism: Each new signed state includes a penalty clause — if either party tries to broadcast an old (outdated) state on-chain, the other party can claim ALL the funds by broadcasting the newer state within a timeout window. This incentivizes honest behavior.
  • Network routing: You don't need a direct channel with every person. Payments route through multiple connected nodes (like routers on the internet). Hash Time-Locked Contracts (HTLCs) ensure each hop in the route is atomically linked — either the full payment succeeds or everything refunds.
  • Closing a channel: Either party broadcasts the most recent signed state to the Bitcoin mainchain. Funds are distributed according to that final state after a timelock period.
  • Impact: Lightning enables Bitcoin micropayments (fraction of a cent), instant settlement, and theoretically millions of TPS — making Bitcoin viable for everyday purchases like coffee, not just large store-of-value transactions.

Compare Proof-of-Work and Proof-of-Stake. Why did Ethereum transition to PoS?

10M▼

  • Finality difference: PoW has probabilistic finality (6 blocks = safe). PoS Ethereum has cryptographic finality — after 2 "checkpoint" epochs (~12.8 minutes), transactions are mathematically impossible to revert without slashing 1/3 of all staked ETH.

Explain Ethereum Account Management — EOA vs Contract Accounts, Transactions, Gas, and Internal Transactions.

10M▼
Ethereum uses an account-based model (unlike Bitcoin's UTXO model). All accounts are "state objects" that hold a balance (in Wei), a nonce (transaction counter), and in the case of Contract Accounts, also code and storage.
Two Types of Ethereum Accounts
FeatureExternally Owned Account (EOA)Contract Account (CA)
Controlled ByA private key held by a human userSmart contract code deployed on the blockchain
Can Initiate TXYes — EOAs start all transactionsNo — can only react to incoming transactions/calls
Has CodeNo — just a balance and nonceYes — contains immutable EVM bytecode
Has StorageNoYes — persistent key-value storage on the blockchain
Gas FeesMust pay gas for all outgoing transactionsGas is paid by the EOA that triggered the chain of calls
ExamplesMetaMask wallets, hardware wallet addressesUniswap contracts, ERC-20 token contracts, DAOs
  • Nonce: Every account has a nonce — a transaction counter. For EOAs, it increments by 1 for each outgoing transaction. This prevents replay attacks (rebroadcasting an old signed transaction) and ensures transactions execute in the correct order.
  • Three types of transactions: (1) Value Transfers — sending ETH from one EOA to another; (2) Contract Creation — deploying new smart contract bytecode to the network, creating a new Contract Account; (3) Contract Interaction — calling a function in an existing contract to execute logic or change state.
  • Gas: Every EVM operation (addition, storage write, hash computation) costs a defined amount of gas units. Users specify: gasLimit (maximum gas willing to spend) and maxFeePerGas (max ETH per gas unit). If execution exceeds gasLimit, it reverts but the gas is still consumed.
  • EIP-1559 Gas Reform (2021): Split fees into Base Fee (burned, reducing ETH supply — deflationary pressure) and Priority Fee (tip to validator for inclusion). The Base Fee adjusts automatically based on network demand — targeting ~50% full blocks.
  • Internal Transactions: When a smart contract calls another contract during execution, those are "internal transactions" (also called "message calls"). They don't appear in the main transaction list but are recorded in transaction traces. A single user-initiated transaction can trigger dozens of internal transactions across multiple contracts.
  • Account Abstraction (ERC-4337, 2023): A new standard allowing Contract Accounts to behave like EOAs — enabling smart contract wallets with features like multi-sig, social recovery, gas sponsorship, and batch transactions. Removes the restriction that only EOAs can initiate transactions.

Explain Error Handling in Smart Contracts — require, revert, assert, custom errors, try/catch.

10M▼
Error handling in Solidity ensures atomicity — if any step in a transaction fails, ALL state changes are rolled back (reverted) to the previous valid state. This prevents partial execution that could leave the blockchain in an inconsistent state.
Mechanism Use Case Gas Behavior Example
require() Input validation, access control, checking preconditions Refunds remaining gas to user require(msg.value > 0, "Must send ETH");
revert() Complex if/else logic where require() is awkward Refunds remaining gas to user if(x != 5) revert InvalidInput();
assert() Internal invariants — conditions that should NEVER be false if code is correct v0.8+: refunds gas. Before: consumed all gas assert(balance >= 0);
Custom Errors (v0.8.4+) Gas-efficient alternative to string error messages Stores 4-byte selector vs full string — much cheaper error Unauthorized(address caller);
try/catch (v0.6+) Handling failures from external contract calls gracefully Allows caller to recover without full revert try token.transfer(to, amt) { ... } catch { ... }
  • require() — The workhorse: Used at the start of functions to validate conditions before executing logic. If the condition is false, it reverts all changes and returns the error message. Remaining gas is refunded to the caller. Example: require(balance[msg.sender] >= amount, "Insufficient balance");
  • revert() — Explicit reversion: Used inside complex branching logic. Functionally identical to require() failing, but allows cleaner code structure. Always refunds remaining gas.
  • assert() — Panic detector: Should only be used for conditions that mathematically cannot be false if the code is correct. If assert fails, it indicates a bug (not a user error). In v0.8+, generates a Panic(0x01) error and refunds remaining gas.
  • Custom Errors (v0.8.4+): Defined outside functions with error ErrorName(params). When used with revert, only a 4-byte function selector is stored instead of a full string — significantly reducing gas costs for deployments and reverts.
  • try/catch: Only works for external calls and contract creation. Allows a contract to call another contract and handle failure without reverting the entire transaction — critical for complex multi-contract interactions in DeFi.
  • Checks-Effects-Interactions pattern: Best practice for preventing reentrancy. Order: (1) Check conditions with require(); (2) Update state variables (effects); (3) Make external calls (interactions). This ensures the contract's state is updated BEFORE calling external code that might call back in.
  • Reentrancy vulnerability: The infamous DAO hack (2016) exploited a pattern where ETH was sent (interaction) before the balance was updated (effect). A malicious contract receiving ETH called back into the sender, which still showed the full balance — draining millions.

Q30

Discuss Best Practices and Auditing for Smart Contracts.

10M▼
Smart contract security is critical because deployed contracts manage billions in value and are often immutable — bugs cannot be patched after deployment. A single vulnerability can result in total loss of funds with no recourse. Rigorous development practices and independent auditing are non-negotiable.
Development Best Practices
  • Use Proven Libraries (OpenZeppelin): Never reinvent cryptographic primitives or token standards. OpenZeppelin provides audited, battle-tested implementations of ERC-20, ERC-721, Access Control, and more. Using them eliminates entire classes of common vulnerabilities.
  • Checks-Effects-Interactions Pattern: Always update state variables BEFORE making external calls. This prevents reentrancy attacks — the pattern ensures that even if an external contract calls back, the state is already updated correctly.
  • Keep It Simple (KISS): Complex code has more attack surface. Break large contracts into smaller, modular contracts with single responsibilities. Simple code is easier to audit and harder to exploit.
  • Input Validation: Sanitize and validate all user-controlled inputs. Use require() at function entrances to check amounts, addresses, and access permissions before executing any logic.
  • Upgradability via Proxy Patterns: Since contracts are immutable after deployment, use OpenZeppelin's upgradeable proxy pattern (UUPS or Transparent Proxy) to allow bug fixes without migrating funds — but ensure the proxy's admin keys are secured with a timelock or multisig.
  • Comprehensive Testing: Use Hardhat or Truffle frameworks to write unit tests covering all paths, edge cases, and attack scenarios. Target 100% code coverage. Include fuzz testing (random input testing) to discover unexpected behaviors.
The Audit Process
  • Scoping: Auditors review code scope, architecture diagrams, and any known risks flagged by the development team. They understand the business logic to spot "logic errors" — correct code implementing wrong business rules.
  • Automated Analysis: Tools like Slither (Python-based static analyzer), Mythril, and Manticore scan code for known vulnerability patterns: reentrancy, integer overflow, unchecked external calls, access control issues.
  • Manual Line-by-Line Review: The most critical phase — auditors trace every possible execution path, check mathematical formulas for overflow, verify every access control gate, and model attack scenarios.
  • Remediation & Re-audit: The development team fixes all found issues. The auditor verifies each fix is correct and doesn't introduce new problems. For high-value protocols, two separate audit firms are recommended.
